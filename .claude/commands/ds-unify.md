Ти - експерт з дизайн-систем та CSS-уніфікації для проекту IPRM (Flask/Jinja2/CSS).

## Мета

Аудит шаблонів та CSS на предмет **уніфікації стилів з дизайн-системою**. Знайти елементи, що візуально дублюють компоненти дизайн-системи, але використовують власні (кастомні) стилі замість стандартних. Привести до єдиного вигляду. Також виявити нові корисні компоненти, яких бракує в дизайн-системі.

> **Зона відповідальності:** відповідність візуальних елементів шаблонів компонентам дизайн-системи.
> **НЕ перевіряй:** Python-код, структуру каталогів, CI/CD -- це зони `/audit-structure` та `/css-split`.

$ARGUMENTS

Якщо аргументи не передано -- провести повний аудит всіх шаблонів.

## Що таке дизайн-система проекту

Дизайн-система -- це набір стандартних компонентів, визначених у:
- `app/static/css/common.css` -- CSS-класи для всіх публічних компонентів
- `app/static/css/admin.css` -- CSS-класи для адмін-компонентів
- `app/static/css/auth.css` -- CSS-класи для авторизації
- `app/templates/design_system/_content.html` -- каталог компонентів з прикладами використання
- `app/templates/design_system/index.html` -- навігація по компонентах

## Крок 1: Побудувати реєстр дизайн-системи

Прочитай `common.css`, `admin.css`, `auth.css` та `design_system/_content.html`.

Для кожного компонента зафіксуй:
- **Назва** та CSS-клас (напр. "Primary Button" -> `.iprm-btn.iprm-btn--primary`)
- **Ключові стилі**: розмір шрифту, padding, border-radius, background, color
- **Варіанти/модифікатори**: `--sm`, `--primary`, `--disabled` тощо
- **Контекст**: де задокументований -- в `_content.html` чи лише в CSS

Категоризуй компоненти:

### Публічна частина (iprm-*)
- Кнопки: `.iprm-btn`, `.iprm-btn--primary`, `.iprm-btn--sm`, `.iprm-btn--disabled`
- Теги: `.iprm-tag`
- Картки: `.iprm-card`, `.iprm-photo-card`, `.iprm-course-card`
- Hero: `.iprm-hero`, `.iprm-hero__title`, `.iprm-hero__subtitle`, `.iprm-hero__price`
- Секції: `.iprm-section`, `.iprm-section__title`, `.iprm-section__subtitle`, `.iprm-section__cta`
- Тренер: `.iprm-trainer`, `.iprm-trainer__photo`, `.iprm-trainer__name`, `.iprm-trainer__role`
- Програма: `.iprm-program`, `.iprm-program__block`
- Target grid: `.iprm-target-grid`, `.iprm-target-card`
- Alerts: `.alert`, `.alert--error`, `.alert--success`, `.alert--info`
- Footer: `.iprm-footer`
- Header/Nav: `.iprm-header`, `.iprm-nav`

### Адмін-частина (admin-*)
- Таблиці, форми, кнопки, модалки, бейджі та інші елементи з `admin.css`

### Авторизація (auth-*)
- Форми, кнопки, повідомлення та інші елементи з `auth.css`

## Крок 2: Сканування шаблонів

Прочитай ВСІ шаблони з `app/templates/` (рекурсивно, включаючи partials).
НЕ читай шаблони з `design_system/` -- це довідкова сторінка.

Для кожного шаблону знайди ВСІ HTML-елементи та порівняй з реєстром дизайн-системи.

### 2.1 Перевірка кнопок

Знайди всі елементи, що виглядають як кнопки:
- `<a>`, `<button>`, `<input type="submit">` з CSS-класами
- Порівняй їх класи зі стандартними `.iprm-btn`, `.admin-btn`, `.auth-btn`
- Позначити як ISSUE, якщо:
  - Кнопка має кастомний клас, але виглядає як стандартна (той самий padding, border-radius, bg)
  - Кнопка має inline style замість класу
  - Кнопка має клас, що визначений в page-*.css, але візуально ідентична стандартній

### 2.2 Перевірка таблиць

Знайди всі `<table>` елементи:
- Чи використовують стандартні класи з дизайн-системи?
- Чи однаковий стиль заголовків, рядків, обводок?
- Позначити як ISSUE, якщо різні таблиці мають різні стилі для однакових цілей

### 2.3 Перевірка заголовків

Знайди всі `<h1>`-`<h6>`:
- Чи відповідають типографічній шкалі дизайн-системи?
- Чи використовують стандартні класи (`.iprm-section__title`, `.iprm-hero__title` тощо)?
- Позначити як ISSUE, якщо заголовок має кастомний розмір, що не відповідає жодному рівню шкали

### 2.4 Перевірка форм

Знайди всі `<form>` та їх елементи:
- `<input>`, `<select>`, `<textarea>`, `<label>`
- Чи використовують стандартні класи?
- Чи однаковий стиль полів у різних формах?

### 2.5 Перевірка карток

Знайди всі елементи, що виглядають як картки (bg, border-radius, padding):
- Чи використовують стандартні класи?
- Чи відповідають вже існуючим компонентам?
- Позначити як ISSUE, якщо картка має кастомний стиль, що дублює `.iprm-card`, `.iprm-photo-card` тощо

### 2.6 Перевірка бейджів/статусів/тегів

Знайди всі маленькі елементи-мітки:
- Бейджі статусу, теги, лейбли
- Чи використовують стандартні класи (`.iprm-tag`, `.badge`, `.admin-badge`)?
- Позначити як ISSUE, якщо є кастомний бейдж

### 2.7 Перевірка алертів/повідомлень

Знайди всі повідомлення (flash messages, help text, error messages):
- Чи використовують `.alert`, `.alert--error`, `.alert--success`?

### 2.8 Перевірка іконок та зображень

- Чи мають placeholder-и однаковий стиль? (`dashed border`, `var(--iprm-placeholder-grad)`)
- Чи відповідають aspect-ratio стандартним компонентам?

## Крок 3: Сканування CSS

Для кожного page-*.css та контекстного CSS (admin.css, auth.css, registration.css):

### 3.1 Знайти дублювання стилів

Порівняй CSS-правила з `common.css`. Для кожного класу в page-*.css перевір:
- Чи є в common.css клас з такими ж або дуже схожими стилями (+-2px, той самий колір)?
- Якщо так -- це дублювання, ISSUE

### 3.2 Знайти візуальні дублікати

CSS-класи з різними іменами, але однаковими стилями:
- Однаковий `font-size` + `font-weight` + `color` = дублювання типографіки
- Однаковий `padding` + `border-radius` + `background` = дублювання кнопки/картки
- Однаковий `border` + `border-radius` + стилі таблиці = дублювання таблиці

### 3.3 Знайти відхилення від CSS-змінних

Перевір, чи використовують page-*.css стандартні CSS-змінні:
- Кольори: `var(--iprm-text)`, `var(--iprm-dark)`, `var(--iprm-accent)` тощо
- Радіуси: `var(--iprm-radius)`, `var(--iprm-radius-sm)`
- Фони: `var(--iprm-card-bg)`, `var(--iprm-placeholder-grad)`
- Позначити як WARNING, якщо хардкоджене значення замість змінної

## Крок 4: Виявлення нових компонентів

Знайди елементи, яких НЕМАЄ в дизайн-системі, але які:
- Використовуються на 2+ сторінках
- Мають стабільний, повторюваний вигляд
- Логічно є окремим компонентом (бейдж, попап, tooltip, breadcrumb, pagination тощо)

Для кожного такого елемента:
- Назва компонента
- Де використовується (шаблони)
- Поточний CSS (файл:рядок)
- Рекомендація: додати до дизайн-системи

## Крок 6: Виправлення

### Порядок виправлень

1. **Спочатку -- порушення уніфікації** (від найбільш критичних до найменших):
   - Замінити кастомні класи на стандартні в шаблонах
   - Видалити непотрібні кастомні CSS-правила
   - Замінити хардкоджені значення на CSS-змінні

2. **Потім -- нові компоненти**:
   - Визначити стандартний CSS-клас (з BEM-іменуванням)
   - Додати стилі у відповідний CSS-файл (common.css для публічних, admin.css для адмін)
   - Додати приклад у `design_system/_content.html`
   - Додати посилання в навігацію `design_system/index.html`
   - Замінити кастомні класи у шаблонах на нові стандартні

### Правила заміни класів

При заміні кастомного класу на стандартний:
1. Перевір, що стандартний клас повністю покриває потрібний вигляд
2. Якщо потрібна невелика варіація -- додай модифікатор (`--modifier`), а не новий клас
3. Видали замінений кастомний CSS тільки якщо він більше ніде не використовується
4. Перевір responsive-стилі -- вони мають бути у стандартному компоненті

### Правила додавання нового компонента до DS

При додаванні нового компонента:
1. CSS-клас має відповідати BEM-конвенції проекту:
   - Публічний: `.iprm-{block}__{element}--{modifier}`
   - Адмін: `.admin-{block}__{element}--{modifier}`
2. Використовуй лише CSS-змінні з `:root` (не хардкоджені значення)
3. Додай responsive-стилі (@media 768px, 480px)
4. Додай приклад в `_content.html` за шаблоном:
```html
<div class="ds-section" id="{component-id}">
  <p class="ds-label">Components</p>
  <h2 class="ds-title">{Назва компонента}</h2>
  <!-- приклад компонента -->
  <p class="ds-hint">{опис ключових стилів}</p>
</div>
```
5. Додай посилання в навігацію `index.html`

## Крок 7: Верифікація

Після виправлень перевір:
- Всі замінені елементи виглядають так само як стандартні
- Жоден кастомний стиль не загублений (якщо він не дублював стандартний)
- Нові компоненти правильно відображаються на сторінці дизайн-системи
- Responsive-стилі працюють для нових компонентів
- Навігація дизайн-системи оновлена

## Обмеження

- НЕ змінюй CSS-змінні в `:root` або `[data-theme]`
- НЕ видаляй стилі, що НЕ дублюють дизайн-систему
- НЕ аналізуй Python-код, конфігурацію -- це зона `/audit-structure`
- НЕ аналізуй файлову структуру CSS -- це зона `/css-split`
- НЕ використовуй `!important`
- НЕ змінюй шаблони в `design_system/` при пошуку порушень (тільки при додаванні нових компонентів)
- @media queries додавай/перенось разом з компонентом
- Пиши коментарі українською
- Будь конкретним: вказуй файл:рядок для кожного порушення
