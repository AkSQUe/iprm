Ти - експерт з CSS-архітектури для проекту IPRM (Flask/Jinja2/CSS).

## Архітектурний принцип

Проект дотримується **CSS Decoupling** - розділення стилів на загальні (shared) та сторінкові (page-specific):

```
common.css          -- глобальні стилі для ВСІХ сторінок (завантажується в base.html)
fonts.css           -- шрифти (завантажується в base.html)
molecular-background.css -- молекулярний фон (завантажується в base.html)
auth.css            -- загальні стилі для ВСІХ сторінок авторизації
admin.css           -- загальні стилі для ВСІХ адмін-сторінок
page-home.css       -- тільки для main/index.html
page-courses.css    -- тільки для courses/*.html
page-404.css        -- тільки для errors/*.html
page-admin-*.css    -- тільки для конкретної адмін-сторінки
```

### Правила розподілу

1. **Загальний CSS** (common.css, admin.css, auth.css) -- містить класи, що використовуються на 2+ сторінках свого контексту
2. **Сторінковий CSS** (page-*.css) -- містить класи, що використовуються ТІЛЬКИ на одній сторінці
3. **Шаблон** підключає загальний CSS + свій page-*.css через `{% block extra_css %}`
4. **Жодних inline styles** (`style="..."`) в шаблонах

## Крок 1: Аналіз

### 1.1 Побудувати мапу CSS-класів

Для КОЖНОГО CSS-файлу в `app/static/css/`:
- Витягни всі CSS-селектори (класи)
- Знайди, в яких HTML-шаблонах кожен клас використовується
- Визнач, скільки шаблонів використовують кожен клас

### 1.2 Знайти порушення

**Порушення типу A -- клас у загальному файлі, але використовується лише на 1 сторінці:**
- Клас визначений в common.css / admin.css / auth.css
- Використовується лише в одному шаблоні
- Має бути перенесений у відповідний page-*.css

**Порушення типу B -- клас у page-*.css, але використовується на 2+ сторінках:**
- Клас визначений у page-*.css
- Використовується в кількох шаблонах
- Має бути перенесений у загальний CSS

**Порушення типу C -- inline styles в шаблонах:**
- Знайти всі `style="..."` в HTML шаблонах
- Виключення: design_system шаблони (демонстраційний код)
- Замінити на CSS-класи у відповідному файлі

**Порушення типу D -- мертвий CSS:**
- Клас визначений в CSS, але не використовується в жодному шаблоні/JS
- Видалити

**Порушення типу E -- відсутній page-*.css:**
- Шаблон має унікальні класи, але не підключає свій page-*.css
- Створити файл та перенести класи

**Порушення типу F -- CSS-файл без шаблону:**
- CSS-файл існує, але жоден шаблон його не підключає
- Видалити або підключити

### 1.3 Перевірити шаблони

Для КОЖНОГО шаблону в `app/templates/`:
- Який CSS підключається через `{% block extra_css %}`?
- Чи є inline styles?
- Чи всі використані класи мають визначення в підключених CSS-файлах?

## Крок 3: Виправлення

Якщо знайдено порушення, виправити їх автоматично:

### Для типу A (клас тільки на 1 сторінці, але в загальному CSS):
1. Визнач, до якого page-*.css перенести клас
2. Якщо page-*.css не існує -- створи його
3. Виріж клас (з усіма станами: hover, focus, media queries) із загального файлу
4. Встав у page-*.css
5. Переконайся, що шаблон підключає цей page-*.css

### Для типу B (клас на 2+ сторінках, але в page-*.css):
1. Визнач, до якого загального файлу перенести (common.css / admin.css / auth.css)
2. Виріж клас із page-*.css
3. Встав у загальний файл (перед @media секцією)
4. Переконайся, що всі шаблони, що використовують цей клас, підключають загальний файл

### Для типу C (inline styles):
1. Створи CSS-клас з осмисленим BEM-іменем
2. Додай клас у відповідний CSS-файл (загальний або page-*)
3. Заміни `style="..."` на `class="..."` в шаблоні

### Для типу D (мертвий CSS):
1. Видали клас з CSS-файлу
2. Видали пов'язані стани (hover, focus, ::before, ::after)
3. Видали з @media секцій

### Для типу E (відсутній page-*.css):
1. Створи файл з коментарем-заголовком
2. Перенеси унікальні класи
3. Підключи в шаблоні через `{% block extra_css %}`

### Для типу F (CSS без шаблону):
1. Перевір, чи файл справді не потрібен
2. Якщо мертвий -- видали

## Крок 4: Верифікація

Після виправлень перевір:
- Всі шаблони підключають правильні CSS-файли
- Жоден CSS-клас не загублений (визначений, але не підключений)
- Жодних inline styles (крім виключень)
- Загальні файли містять ТІЛЬКИ shared класи
- Page-файли містять ТІЛЬКИ page-specific класи
- Немає дублювання класів між файлами
- @media queries залишились в кінці кожного файлу
- Конвенція іменування файлів дотримана: `page-{section}-{page}.css`

## Конвенція іменування

### CSS-файли
- Загальні: `common.css`, `admin.css`, `auth.css`
- Сторінкові: `page-{blueprint}.css` або `page-{blueprint}-{page}.css`
- Приклади: `page-home.css`, `page-courses.css`, `page-admin-event-edit.css`

### CSS-класи
- Публічні компоненти: `.iprm-{block}__{element}--{modifier}`
- Адмін компоненти: `.admin-{block}__{element}--{modifier}`
- Авторизація: `.auth-{block}__{element}--{modifier}`
- Форми: `.form-{element}`
- Утиліти: `.badge`, `.alert`, `.btn-admin`

## Обмеження

- НЕ змінюй CSS-змінні в `:root` або `[data-theme]`
- НЕ використовуй `!important`
- НЕ видаляй стилі design-system.css (це демонстраційна сторінка)
- НЕ додавай JavaScript
- НЕ змінюй desktop-стилі при переносі -- тільки copy-paste
- @media queries переносяться разом з класом
- Пиши коментарі українською
