## Мета

Аудит та виправлення **всього frontend-коду**: HTML-шаблони, CSS-стилі, JavaScript. Перевірка якості, архітектури та відповідності принципам проекту.

> **НЕ перевіряй:** Python-код, конфігурацію, CI/CD, моделі -- це зона `/audit-structure`.

## Архітектурний принцип CSS

Проект дотримується **CSS Decoupling** -- розділення стилів на загальні (shared) та сторінкові (page-specific):

```
common.css          -- глобальні стилі для ВСІХ сторінок (завантажується в base.html)
fonts.css           -- шрифти (завантажується в base.html)
molecular-background.css -- молекулярний фон (завантажується в base.html)
auth.css            -- загальні стилі для ВСІХ сторінок авторизації
admin.css           -- загальні стилі для ВСІХ адмін-сторінок
page-home.css       -- тільки для main/index.html
page-courses.css    -- тільки для courses/*.html
page-404.css        -- тільки для errors/*.html
page-admin-*.css    -- тільки для конкретної адмін-сторінки
```

### Правила розподілу CSS

1. **Загальний CSS** (common.css, admin.css, auth.css) -- класи, що використовуються на 2+ сторінках свого контексту
2. **Сторінковий CSS** (page-*.css) -- класи, що використовуються ТІЛЬКИ на одній сторінці
3. **Шаблон** підключає загальний CSS + свій page-*.css через `{% block extra_css %}`
4. **Жодних inline styles** (`style="..."`) в шаблонах

## Крок 1: CSS-аудит

### 1.1 Побудувати мапу CSS-класів

Для КОЖНОГО CSS-файлу в `app/static/css/`:
- Витягни всі CSS-селектори (класи)
- Знайди, в яких HTML-шаблонах кожен клас використовується
- Визнач, скільки шаблонів використовують кожен клас

### 1.2 Знайти CSS-порушення

**[A] Клас у загальному файлі, але на 1 сторінці:**
- Клас визначений в common.css / admin.css / auth.css
- Використовується лише в одному шаблоні
- Виняток: BEM-модифікатори (--modifier) залишаються з блоком

**[B] Клас у page-*.css, але на 2+ сторінках:**
- Клас визначений у page-*.css
- Використовується в кількох шаблонах

**[C] Inline styles в шаблонах:**
- Знайти всі `style="..."` в HTML-шаблонах
- Виключення: design_system шаблони (демонстраційний код)

**[D] Мертвий CSS:**
- Клас визначений в CSS, але не використовується в жодному шаблоні/JS

**[E] Відсутній page-*.css:**
- Шаблон має унікальні класи без власного page-*.css

**[F] CSS-файл без шаблону:**
- CSS-файл існує, але жоден шаблон його не підключає

## Крок 2: HTML-аудит

### 2.1 No Inline Policy
- Знайти всі `style="..."` в шаблонах -> замінити на CSS-класи
- Знайти всі `<script>` блоки (inline JS) -> виділити в зовнішні .js файли
- Допустимі виключення:
  - theme-init скрипт у `base.html` (запобігає FOUC)
  - design_system шаблони (демонстраційний код)

### 2.2 Семантика та структура
- Чи використовуються семантичні теги (`<header>`, `<nav>`, `<main>`, `<footer>`, `<article>`, `<section>`)?
- Чи є `<main>` на кожній сторінці?
- Чи немає порожніх / безсенсових `<div>` wrapper-ів?

### 2.3 Доступність (базова)
- Чи є `lang="uk"` на `<html>`?
- Чи є `<meta name="viewport">` в `base.html`?
- Чи є `alt` атрибути на `<img>`?
- Чи мають форми `<label>` для кожного `<input>`?
- Чи мають інтерактивні елементи достатній touch-розмір (мін. 44x44px)?

## Крок 5: Виправлення

### CSS-виправлення

**Для типу A** (клас на 1 сторінці в загальному файлі):
1. Визнач page-*.css (створи якщо потрібно)
2. Виріж клас з усіма станами (hover, focus, @media)
3. Встав у page-*.css
4. Підключи в шаблоні

**Для типу B** (клас на 2+ сторінках в page-*.css):
1. Виріж клас із page-*.css
2. Встав у загальний файл (перед @media секцією)

**Для типу C** (inline styles):
1. Створи CSS-клас з BEM-іменем
2. Додай у відповідний CSS-файл
3. Заміни `style="..."` на `class="..."`

**Для типу D** (мертвий CSS):
1. Видали клас та пов'язані стани/медіа

### HTML-виправлення

**Inline JS** -> винеси в `app/static/js/{шаблон-назва}.js`, підключи через `<script src="...">`
**Inline CSS** -> створи клас, додай в CSS-файл
**Семантика** -> заміни `<div>` на відповідний семантичний тег

### JS-виправлення

**Мертвий JS** -> видали файл
**Хардкод** -> використай CSS-змінні через `getComputedStyle()`

## Крок 6: Верифікація

Після виправлень перевір:
- Всі шаблони підключають правильні CSS-файли
- Жоден CSS-клас не загублений
- Жодних inline styles (крім виключень)
- Жодного inline JS (крім виключень)
- Загальні CSS містять ТІЛЬКИ shared класи
- Page-CSS містять ТІЛЬКИ page-specific класи
- Немає дублювання класів між файлами
- @media queries залишились в кінці кожного CSS-файлу
- Конвенція іменування файлів дотримана

## Конвенція іменування

### CSS-файли
- Загальні: `common.css`, `admin.css`, `auth.css`
- Сторінкові: `page-{blueprint}.css` або `page-{blueprint}-{page}.css`

### CSS-класи
- Публічні: `.iprm-{block}__{element}--{modifier}`
- Адмін: `.admin-{block}__{element}--{modifier}`
- Авторизація: `.auth-{block}__{element}--{modifier}`
- Форми: `.form-{element}`
- Утиліти: `.badge`, `.alert`, `.btn-admin`

### JS-файли
- Глобальні: `theme-toggle.js`, `molecular-background.js`
- Сторінкові: `admin-event-edit.js`, `admin-trainer-edit.js`

## Обмеження

- НЕ змінюй CSS-змінні в `:root` або `[data-theme]`
- НЕ використовуй `!important`
- НЕ видаляй стилі design-system.css (демонстраційна сторінка)
- НЕ змінюй desktop-стилі при переносі -- тільки copy-paste
- НЕ аналізуй Python-код, конфігурацію, моделі -- це зона `/audit-structure`
- @media queries переносяться разом з класом
- Пиши коментарі українською
