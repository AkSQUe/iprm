## ФАЗА 1: Збір контексту

### 1.1 Визначення scope

На основі переданих аргументів визначи цільовий scope:
- **Файл/модуль**: конкретний `.py`, `.html`, `.css` файл
- **Blueprint/сторінка**: весь blueprint (routes + forms + templates + CSS)
- **Сервіс/функція**: конкретний сервісний шар або feature

### 1.2 Глибоке читання

Прочитай ВСІ файли в scope повністю. Для кожного файлу зафіксуй:
- Призначення та відповідальність
- Публічний інтерфейс (функції, класи, endpoints)
- Залежності (imports, використані моделі, сервіси)
- Зв'язки з іншими частинами (хто викликає, хто залежить)

### 1.3 Збір залежностей

Знайди ВСІ файли, що взаємодіють з цільовою імплементацією:
- `grep` по назвах функцій, класів, endpoints
- Routes, що викликають сервіси
- Templates, що рендеряться з routes
- CSS, що використовується в templates
- Forms, що обробляються в routes
- Models, що використовуються в queries
- Тести, що покривають цю імплементацію

## ФАЗА 2: Аналіз

Для кожного файлу в scope проведи аналіз за категоріями. Кожну знахідку класифікуй:

### 2.1 Помилки та баги (BUG)

- Runtime помилки (uncaught exceptions, missing None-checks)
- Логічні помилки (неправильні умови, off-by-one)
- Race conditions та concurrency проблеми
- Broken edge cases (порожні списки, null values, Unicode, великі числа)
- Security: SQL injection, XSS, CSRF bypass, path traversal, command injection
- Security: відсутня авторизація або неправильна перевірка прав

### 2.2 Edge Cases (EDGE)

- Що станеться при порожньому вводі?
- Що станеться при дуже довгому вводі?
- Що станеться при спеціальних символах?
- Що станеться при одночасних запитах?
- Що станеться при відсутності з'єднання з БД/зовнішнім сервісом?
- Що станеться при timeout зовнішнього API?
- Що станеться при переповненні ліміту?
- Граничні значення: 0, 1, MAX, від'ємні числа, порожні рядки, None

### 2.3 Якість коду (QUALITY)

- DRY: дублювання коду (в рамках scope та між файлами)
- SRP: файл/функція робить забагато
- Надто довгі функції (>30 рядків бізнес-логіки в route)
- Надто глибока вкладеність (>3 рівні)
- Неочевидна логіка без коментарів
- Magic numbers та hardcoded strings
- Невідповідність naming conventions проекту
- Мертвий код (unreachable, unused imports, unused variables)

### 2.4 Продуктивність (PERF)

- N+1 запити до БД
- Відсутні індекси для часто використаних фільтрів/сортувань
- Зайві запити до БД (можна об'єднати або кешувати)
- Великі response без пагінації
- Блокуючі операції (синхронний IO де можна async)
- Надмірна серіалізація/десеріалізація

### 2.5 Надійність (RELIABILITY)

- Error handling: чи обробляються всі можливі помилки?
- Graceful degradation: чи працює система при partial failure?
- Logging: чи логуються важливі операції та помилки?
- Validation: чи валідуються дані на вході (forms, API params)?
- Idempotency: чи безпечні повторні запити?
- Transactions: чи правильно використовуються DB transactions?

### 2.6 Архітектура (ARCH)

- Чи відповідає файлова структура принципам проекту?
- Чи правильно розподілені відповідальності (routes vs services vs models)?
- Чи немає circular dependencies?
- Чи можна модулізувати великий файл?
- Чи дотримано Separation of Concerns?

## ФАЗА 3: Звіт та план

### 3.1 Зведений звіт

Виведи знахідки у форматі:

```
=== ANALYSIS: [назва scope] ===

--- Files analyzed ---
file_path: N lines, purpose

--- Findings ---
[BUG]     file:line -- опис проблеми
[EDGE]    file:line -- опис edge case
[QUALITY] file:line -- опис проблеми якості
[PERF]    file:line -- опис проблеми продуктивності
[RELIABILITY] file:line -- опис проблеми надійності
[ARCH]    file:line -- опис архітектурної проблеми

--- Зведення ---
BUG: N | EDGE: N | QUALITY: N | PERF: N | RELIABILITY: N | ARCH: N
```

### 3.2 План покращень

Згрупуй знахідки у план дій з пріоритетами:

**CRITICAL (зробити обов'язково):**
- Баги, security-проблеми, data integrity issues
- Формат: Проблема -> Рішення -> Файли -> Складність (S/M/L)

**IMPORTANT (сильно рекомендовано):**
- Edge cases, reliability, N+1, відсутня валідація
- Формат: Проблема -> Рішення -> Файли -> Складність (S/M/L)

**IMPROVE (якість та підтримуваність):**
- DRY, SRP, модулізація, naming, мертвий код
- Формат: Проблема -> Рішення -> Файли -> Складність (S/M/L)

### 3.3 Пропозиції розширення функціоналу

На основі аналізу запропонуй логічні розширення, що покращать UX та функціональність:
- Що очікує користувач, але ще не реалізовано?
- Які суміжні функції були б корисними?
- Які автоматизації можна додати?
- Які покращення UX були б доречними?

Формат:
```
--- Можливі розширення ---
[EXTEND] Назва -- опис, обгрунтування, оцінка складності (S/M/L)
```

### 3.4 Отримати затвердження

Показати повний план користувачу та чекати підтвердження. Користувач може:
- Затвердити весь план
- Вибрати окремі пункти
- Додати свої побажання
- Відхилити та попросити переаналізувати

## ФАЗА 4: Реалізація

Після затвердження плану реалізуй покращення послідовно:

### 4.1 Порядок реалізації

1. **CRITICAL** -- спочатку фікси безпеки та багів
2. **IMPORTANT** -- edge cases, validation, reliability
3. **IMPROVE** -- рефакторинг та якість коду
4. **EXTEND** -- нові функції (тільки якщо затверджені)

### 4.2 Для кожного покращення

1. Опиши що саме змінюється та чому
2. Внеси зміни в цільовий файл
3. Негайно оновити ВСІ залежні файли (каскад):
   - Якщо змінив model -> оновити routes, forms, templates, tests
   - Якщо змінив route -> оновити templates, tests
   - Якщо змінив CSS class -> оновити templates що його використовують
   - Якщо змінив function signature -> оновити всі callers
4. Позначити крок як виконаний

### 4.3 Принципи реалізації

- Кожна зміна залишає додаток у робочому стані
- НЕ додавай фреймворків/бібліотек без обгрунтування
- НЕ over-engineer -- мінімально необхідна складність
- НЕ додавай docstrings/коментарі до коду, який не змінювався
- НЕ використовуй inline стилі або скрипти
- Дотримуйся конвенцій проекту (naming, структура, CSS Decoupling)
- Валідуй на вході, довіряй внутрішньому коду
- Логуй важливі операції через `current_app.logger`
