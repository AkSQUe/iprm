Аудит **файлової структури та Python-архітектури** проекту IPRM. Тільки backend, конфігурація, blueprints, моделі, CI/CD, документація.

> **Зона відповідальності:** `.py` файли, структура каталогів, конфігурація, CI/CD, документація.

$ARGUMENTS

Якщо аргументи не передано - провести повний аудит всієї структури.

## Контекст проекту

- Тип: Flask 3.0+ Application Factory + Blueprint архітектура
- ORM: SQLAlchemy, міграції через Flask-Migrate/Alembic
- Frontend: Jinja2 + CSS (гібридна архітектура)
- БД: SQLite (dev), PostgreSQL (prod, pg8000)
- Деплой: GitHub Actions + rsync на VPS + systemd + gunicorn
- Документація архітектури: `docs/architecture.md`
- Принципи проекту описані в CLAUDE.md

## Крок 2: Аудит за категоріями

Перевір кожну категорію і оціни стан: OK / WARNING / ISSUE.

### 2.1 Application Factory та точка входу
- `run.py` - чи правильно викликає `create_app()`?
- `app/__init__.py` - чи використовує Application Factory pattern?
- Чи винесені розширення в `extensions.py`?
- Чи немає circular imports?
- Чи є security headers у `after_request` (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, CSP)?
- Чи коректно налаштований Flask-Limiter (rate limiting)?

### 2.2 Blueprint-архітектура
- Динамічно знайди ВСІ каталоги в `app/` що містять `__init__.py` з `Blueprint(` -- це blueprints
- Кожен blueprint: чи має `__init__.py` з реєстрацією Blueprint?
- Чи зареєстровані всі знайдені blueprints в `create_app()`?
- Чи використовує `routes.py` (не `views.py`)?
- Чи є зайві файли всередині blueprint-каталогів?
- Чи дотримано принцип Single Responsibility?
- Чи немає "товстих" routes (бізнес-логіка має бути в services/models)?
- Чи правильно налаштовані url_prefix для кожного blueprint?

### 2.3 Моделі та база даних
- Чи розділені моделі на окремі файли (не один великий `models.py`)?
- Чи є `__init__.py` з правильними імпортами ВСІХ моделей?
- Чи використовується TimestampMixin?
- Чи правильна структура `migrations/`?
- Чи немає міграцій, що конфліктують?
- Порівняй кількість моделей в `app/models/` з імпортами в `app/models/__init__.py` -- чи немає забутих моделей?
- Порівняй моделі з міграціями -- чи всі таблиці мають відповідні міграції?

### 2.4 Файлова структура статики
Перевіряється ЛИШЕ наявність та іменування файлів, НЕ їх вміст:
- CSS: чи відповідає конвенції іменування (common, page-*, blueprint-name)?
- JS: чи відповідає конвенції іменування?
- SVG/зображення: чи організовані по підкаталогах?
- Шрифти: чи є `fonts.css` та відповідні WOFF2 файли?
- Чи немає невикористаних CSS/JS файлів? (порівняй з `url_for('static', ...)` у шаблонах)

### 2.5 Файлова структура шаблонів
Перевіряється ЛИШЕ структура каталогів та зв'язок routes<->templates, НЕ вміст HTML:
- Чи є `base.html` з базовим layout?
- Чи кожен blueprint має свій каталог шаблонів?
- Чи використовуються `partials/` для reusable компонентів?
- Маршрут-шаблон відповідність: перевір що кожен `render_template()` в routes.py посилається на існуючий шаблон
- Зворотна перевірка: чи немає "осиротілих" шаблонів, на які не посилається жоден маршрут?
- Чи немає hardcoded сторінок там, де мають бути динамічні?

### 2.6 Конфігурація та безпека
- `config.py` - чи є Dev/Prod/Test конфігурації?
- `.env` та `.env.example` - чи є приклад для нових розробників?
- `.gitignore` - чи виключені: `.env`, `instance/`, `__pycache__/`, `*.pyc`?
- Git tracking: чи немає `__pycache__` каталогів, що вже відстежуються в git? (перевір через `git ls-files '*__pycache__*'`)
- Чи немає секретів у tracked файлах? (перевір `git ls-files` на наявність .env, *.key, *.pem)
- `keys/` каталог - чи в `.gitignore`?
- Чи є fallback SECRET_KEY в config.py що небезпечний для production?
- Чи валідує ProductionConfig наявність обов'язкових змінних?

### 2.7 CI/CD та DevOps
- `.github/workflows/` - чи є pipeline?
- Чи є `requirements.txt` або `pyproject.toml`?
- Чи є залежності з точними версіями (pinned) чи тільки з діапазонами?
- Чи є `Procfile`, `wsgi.py`, або інший файл для production entry point?
- Чи pipeline виконує тести перед деплоєм?
- Чи pipeline виконує міграції при деплої?

### 2.8 Документація
- Чи є `README.md` з описом проекту?
- Чи є `docs/` каталог з архітектурною документацією?
- Чи відповідає документована структура реальній? (порівняй дерево з `docs/architecture.md` з реальним `app/`)
- Чи документовані ВСІ blueprints? (порівняй список в docs vs реальні каталоги)
- Чи документовані ВСІ моделі? (порівняй з `app/models/`)
- Чи документовані ВСІ маршрути? (порівняй з `docs/routes.md`)

### 2.9 Архітектурні принципи (Python-рівень)
- **DRY**: чи немає дублювання коду між blueprints? (подібні routes, forms)
- **KISS**: чи немає overengineering?
- **SOLID / SRP**: чи кожен Python-файл має одну відповідальність?
- **Separation of Concerns**: чи розділені шари (routes/models/services)?

### 2.10 Відсутні елементи (Flask best practices)
Перевір, чи є потреба в кожному елементі, виходячи з реального стану проекту:
- `app/services/` - шар бізнес-логіки (перевір: чи є routes довші за 30 рядків з бізнес-логікою?)
- `app/utils/` або `app/utils.py` - спільні утиліти (перевір: чи вже існує, що містить?)
- `tests/` каталог з тестами
- `wsgi.py` - WSGI entry point для production (перевір: чи gunicorn використовує run.py чи wsgi?)
- `app/cli.py` - CLI команди (перевір: чи вже існує?)
- Logging конфігурація для production
